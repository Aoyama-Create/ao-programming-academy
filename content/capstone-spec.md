## 1. 対象顧客・課題・解決策

- **対象顧客（ペルソナ）**  
  中小企業の営業部門・経理部門。受注を「申請 → 承認」で回しているが、紙や Excel で管理しており、誰がいつ承認したかの記録が残りにくい。
- **課題**  
  申請・承認のトレーサビリティがない、承認遅れの把握が難しい、データの一元管理ができず CSV で他システムと連携しづらい。
- **解決策**  
  受注の「下書き → 申請 → 承認／却下」を一貫して扱う Web アプリ（B2B 業務アプリ）。ログイン（認証）によりユーザーを識別し、ロールに応じた画面・API の制御（認可）を行う。重要操作の監査ログ、CSV 入出力、承認通知などの非同期処理、本番相当環境へのデプロイまで含める。

---

## 2. 推奨テーマ：受注申請・承認システム（つくるアプリの全機能）

上記を具体化した、**作成する 1 つのアプリ**の定義。以下の機能をすべて備えた受注申請・承認システムとする。

- **認証**：ログイン・セッションまたは API トークンによりユーザーを識別する
- 受注（オーダー）の登録・編集・一覧・詳細（下書き〜確定まで）
- 状態遷移：下書き → 申請 → 承認済み／却下（申請・承認・却下操作）
- ロール（例：申請者・承認者・管理者）に応じた画面・API の制御（RBAC）
- 申請・承認・却下・CSV 出力などの重要操作の監査ログ
- 受注一覧の CSV エクスポート、取引先マスタ等の CSV インポート（UTF-8 等考慮）
- 承認時の通知メール等をキューで非同期実行（Laravel Queue 等）
- 本番相当環境へのデプロイ（デプロイ先は「4.3 デプロイ先」を参照。無料枠の利用可）
- 環境のコード化（Terraform 等の IaC）と CI/CD パイプライン
- （推奨）脅威モデリング・監視設計

推奨テーマで開発する場合は、下記「要件と機能の対応」のとおりで要件を満たす。

## 3. 要件と機能の対応

必須・推奨の各要件について、アプリでは次の機能・画面・処理で満たす。

| 要件 | アプリで実現する機能・画面・処理 |
|------|----------------------------------|
| 認証 | ログイン・セッションまたは API トークンによりユーザーを識別する |
| B2B 業務アプリ | 受注申請・承認システムとして上記機能を提供する企業向け Web アプリ |
| 承認フロー | 受注の状態遷移（下書き → 申請 → 承認/却下）と申請・承認・却下操作 |
| 権限管理（RBAC） | ロール（申請者・承認者・管理者）に応じた画面表示・API 認可 |
| 監査ログ | 申請・承認・却下・CSV 出力等の「誰が・いつ・何を」を記録 |
| CSV 連携 | 受注一覧のエクスポート、マスタのインポート（UTF-8 等考慮） |
| 非同期ジョブ | 承認通知メールのキュー送信、必要に応じて集計バッチ等 |
| デプロイ | 本番相当環境へのデプロイ（AWS ECS / Railway / Render / Fly.io 等） |
| IaC | Terraform 等で環境をコード化し再現可能に |
| CI/CD | テスト・ビルド・デプロイのパイプライン化 |
| 脅威モデリング（推奨） | 上記アプリの脅威モデル（STRIDE 等）を文書化 |
| 監視（推奨） | メトリクス・アラート・ダッシュボードの設計と実装 |

## 4. 技術スタック・DB 構成・デプロイ先

受講生が迷わず着手できるよう、推奨テーマで用いる技術とデータ構造・デプロイ先を以下に固定する。

### 4.1 技術スタック（推奨）

| 区分 | 技術 | 備考 |
|------|------|------|
| フロントエンド | React + TypeScript（Vite） | カリキュラム Stage 3 と同一。ルーティングは React Router 等でよい。 |
| バックエンド | Laravel（PHP 8.x） | カリキュラム Stage 4 と同一。API は REST または Laravel の慣習に沿う。 |
| 認証 | Laravel Breeze または Sanctum | ログイン・セッション・API トークン発行。 |
| データベース | MySQL 8.x または PostgreSQL | いずれか 1 つ。Laravel のマイグレーションで管理する。 |
| キュー | Laravel Queue（driver: database または Redis） | 非同期ジョブ用。Redis はキャッシュ・セッション兼用可。 |
| IaC | Terraform（AWS の場合） | 他クラウドの場合はそのプロバイダでコード化する。 |
| CI/CD | GitHub Actions または GitLab CI | テスト・ビルド・デプロイのパイプライン化。 |

上記以外（別言語・別フレームワーク）で同一要件を満たす実装も可とするが、評価・サポートは推奨スタックを前提とする。

### 4.2 必要な DB 構成（最低限のテーブル・カラム）

受注申請・承認システムで必須となるテーブルと主要カラムを以下に示す。これ以外のカラム（例：受注の金額・日付・取引先 ID）は受講生が設計してよい。

| テーブル | 主なカラム | 役割 |
|----------|------------|------|
| **users** | id, name, email, password, role_id, created_at, updated_at | ユーザーとロールの紐付け。認証・認可の前提。 |
| **roles** | id, name（例: applicant, approver, admin）, created_at, updated_at | RBAC 用。申請者・承認者・管理者などを定義。 |
| **orders** | id, user_id（申請者）, status, created_at, updated_at | 受注。status は draft / pending / approved / rejected 等の enum または文字列。 |
| **audit_logs** | id, user_id, action, target_type, target_id, created_at | 誰が・いつ・何をしたかの記録。申請・承認・却下・CSV 出力等を記録。 |
| **customers**（任意） | id, name, code 等 | 取引先マスタ。CSV インポート対象として使う場合に用意。 |

- **orders** には業務に必要な項目（受注日、金額、品目、取引先 ID 等）を追加してよい。
- **監査ログ**は上記の audit_logs に書き、action で「order.submit」「order.approve」「order.reject」「export.csv」などを識別できるようにする。
- キュー driver に **database** を使う場合は、Laravel 標準の **jobs** テーブル（failed_jobs 含む）を用意する。

### 4.3 デプロイ先（本番相当・無料枠を想定）

本番相当の環境へデプロイする要件を満たすため、以下いずれか（または同等）を利用する。**無料で実施したい場合は 4.3.1 を推奨**する。

- **4.3.1 無料枠で実施する場合（推奨）**  
  - **Railway**（月額クレジット無料枠あり）、**Render**（無料枠あり・スリープあり）、**Fly.io**（無料枠あり）のいずれかで、API・DB・キュー（必要なら）をデプロイする。  
  - フロントは **Vercel** や **Netlify** の無料枠でホスティングし、API の URL を環境変数で指定する構成でよい。  
  - 無料枠の制限（スリープ・リソース上限）は各サービスの最新ドキュメントを確認すること。

- **4.3.2 AWS で実施する場合**  
  - **ECS（Fargate）** または **EC2** にコンテナ／アプリをデプロイし、**RDS** で MySQL/PostgreSQL を利用する。  
  - 無料枠は 12 ヶ月の新規アカウント向けなど制限があるため、コストが発生する可能性がある。予算に応じてリソースサイズを選ぶこと。  
  - IaC は **Terraform** で VPC・ECS・RDS 等を定義し、CI から `terraform plan` / `apply` を実行できるようにする。

提出物では、利用したデプロイ先と「環境の再現手順（README または Terraform の説明）」を明記する。

## 7. 開発の流れ（フェーズ）

1. **要件定義**：対象顧客・課題・解決策の確認、受入基準（Given-When-Then）の整理
2. **設計**：ER 図・主要シーケンス図・RBAC 設計・監査ログの記録対象
3. **実装**：API・フロント・認証・承認フロー・CSV・非同期ジョブ・監査ログ
4. **デプロイ**：IaC で環境構築、CI/CD でテスト・ビルド・デプロイ
5. **監視（推奨）**：メトリクス・アラート・ダッシュボード

週単位の詳細は [週単位カリキュラム - Final：Capstone](/weekly#final-capstone) を参照。
