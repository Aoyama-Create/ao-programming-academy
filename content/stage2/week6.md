**目安時間**: 約 10 時間（解説・テンプレート・例に沿った記述）

## 学習目標

この週が終わると、次のことができるようになります。

- 小さな**要件**を「誰が・何を・なぜ」の形で書ける

- **境界値**・エラーパターンを意識した設計ができる

## 1. 要件定義（小さな機能単位）

要件とは「何を作るか」「なぜ必要か」を、関係者が共有するために文章にしたものです。小さな機能単位で書くと、実装やテストがしやすくなります。

### 誰が・何を・なぜ

次の 3 つをはっきりさせると、要件が伝わりやすくなります。

| 観点 | 説明 | 例 |
|------|------|-----|
| **誰が** | 対象となるユーザーや役割 | 「CLI を使う人」「ログイン済みのユーザー」 |
| **何を** | できるようにしたいこと | 「2 つの数を入力して合計を表示する」 |
| **なぜ** | その機能が必要な理由 | 「手軽に計算結果を確認するため」 |

### 記述例（Stage 1 の CLI の場合）

- **誰が**: ターミナルを使う人
- **何を**: コマンドラインから 2 つの数と演算子を指定し、計算結果を表示する
- **なぜ**: ブラウザや電卓アプリを開かずに、素早く計算したいため

## 2. 受入基準（Given-When-Then）

**受入基準**は「この条件を満たせば、その機能は受け入れられる」と判断するための基準です。**Given-When-Then** の形で書くと、何を前提に・何をしたときに・どうなるべきかが明確になります。

### 書き方

- **Given（前提）**: 状況や初期状態
- **When（操作）**: ユーザーやシステムが行うこと
- **Then（結果）**: 期待される結果

### 例：Stage 1 の簡易計算機

```gherkin
Given ターミナルで calc.js が実行できる環境である
When ユーザーが "node calc.js 10 + 20" を実行する
Then 標準出力に "30" が表示される
```

```gherkin
Given ターミナルで calc.js が実行できる環境である
When ユーザーが "node calc.js 5 * 4" を実行する
Then 標準出力に "20" が表示される
```

```gherkin
Given ターミナルで calc.js が実行できる環境である
When ユーザーが引数なしで "node calc.js" を実行する
Then 使い方（例）が表示される
```

このように、正常系・異常系ごとに「Given-When-Then」を書いておくと、実装の目標とテストの観点がはっきりします。

## 3. 境界設計（入力の範囲・境界値・異常系）

**境界設計**では、入力の「あり得る範囲」や「境界値」「エラーパターン」を意識して設計します。

### 境界値の例

- 数値入力: 最小値・最大値・0・負の数
- 文字数: 空文字・1 文字・上限いっぱい
- 個数: 0 個・1 個・多数

### 異常系の例

- 想定外の引数（演算子が `+` / `-` / `*` / `/` 以外）
- 引数不足・引数过多
- 数値以外が渡された（`node calc.js a + b` など）

### 洗い出しのコツ

- 「正常に動くとき」だけでなく、「変な入力のときどうするか」を 1 つずつ考える
- エラーメッセージや終了コードをどうするかも、受入基準に書いておくとよい

## 4. AI 利用ポリシー

AI（LLM やコード補助ツール）を開発に使うとき、「どこまで任せるか」「どこは人間が必ず確認するか」を決めておくのが **AI 利用ポリシー** です。

### 任せてよい範囲の例

- ボイラープレート（よくある書き方のひな形）
- 既存コードのスタイルに合わせたリファクタの提案
- ドキュメントやコメントの下書き
- テストケースの例やシナリオの下書き

### 必ず人間が検証する領域の例

- **認証・認可**（誰が何をしてよいか）
- **決済**・お金が動く処理
- **暗号化・パスワード・秘密情報**の扱い
- **セキュリティ**（入力検証、インジェクション、権限）
- 法律や規制に関わる処理

### ポリシーの書き方の例

「検証を誰がどうするか」まで書くと、運用しやすくなります。

```markdown
## AI 利用ポリシー（例）

### 任せてよい範囲
- 定型文・README・コメントの下書き
- 既存パターンに沿ったテストケース案

### 必ず人間が行うこと
- 認証・認可・決済・秘密情報を扱うコードのレビューとテスト
- 本番環境への反映前の最終確認

### 検証のしかた
- AI が生成したコードは必ず実行し、意図どおり動くか確認する
- 上記「必ず人間が行うこと」に該当する箇所は、別途チェックリストで確認する
```
